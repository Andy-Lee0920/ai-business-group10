## MVP 프론트엔드 아키텍처 및 코드 리뷰 (미션팟)

현재 구현된 싱글 페이지 형태의 MVP(미션팟) 프론트엔드 코드를 분석한 내용입니다. 빠른 프로토타이핑과 초기 가설 검증을 위해 빠르게 작성된 코드로 보이나, 프로덕트의 확장과 안정적인 운영을 위해 아키텍처 전반에 걸친 개선이 필요합니다. 

시니어 엔지니어 관점에서 즉각적으로 해결해야 할 핵심 페인 포인트와 개선 방향을 정리했습니다.

---

### 1. Separation of Concerns (관심사 분리)

**현상:**
모든 UI 요소(Nav, Hero, Ticker, How-to, Footer, Modal)와 비즈니스 로직(Intersection Observer 애니메이션, 상태 관리)이 하나의 거대한 무명 컴포넌트(`function x()`) 안에 강하게 결합되어 있습니다.

**문제점:**
*   코드 가독성이 매우 떨어지며, 특정 UI나 로직을 수정할 때 예기치 않은 사이드 이펙트가 발생할 확률이 높습니다.
*   컴포넌트 재사용이 불가능합니다.

**개선 제안:**
*   **컴포넌트 모듈화:** `<Navbar>`, `<HeroSection>`, `<MissionModal>`, `<Footer>` 등 독립적인 단위로 컴포넌트를 분할하세요.
*   **비즈니스 로직 분리:** 뷰(View)와 로직(Logic)을 분리해야 합니다. 예를 들어, 스크롤 애니메이션을 담당하는 Intersection Observer 로직은 `useScrollAnimation` 같은 **Custom Hook**으로 추출하여 UI 컴포넌트에서 순수하게 호출만 하도록 구성해야 합니다.

### 2. Error Handling (에러 처리 및 예외 상황 대비)

**현상:**
현재 코드에는 예외 처리 로직이 전무합니다. 모의(Mock) 결제 로직이 단순 `setTimeout`과 `alert`로 구현되어 있으며, 실패 상황에 대한 대비가 없습니다.

**문제점:**
*   추후 실제 API 연동이나 결제 모듈(PG) 연동 시 네트워크 지연, 타임아웃, 서버 에러 등에 대한 Fallback UI가 없어 사용자 경험(UX)이 심각하게 훼손될 수 있습니다.
*   컴포넌트 내부에서 예외가 발생할 경우 React 앱 전체가 백지화(White screen of death)될 위험이 있습니다.

**개선 제안:**
*   **Error Boundary 도입:** 루트 수준 또는 주요 컴포넌트 단위로 `<ErrorBoundary>`를 감싸, 런타임 에러 발생 시 우회할 수 있는 Fallback UI를 제공하세요.
*   **비동기 로직 예외 처리:** 향후 결제 및 데이터 패칭 로직 추가 시 반드시 `try-catch` 블록을 사용하고, 사용자에게 `alert` 대신 비동기 상태(Loading, Success, Error)를 반영하는 토스트(Toast)나 인라인 에러 메시지 UI를 구현해야 합니다.

### 3. Observability (관측 가능성)

**현상:**
유저 인터랙션 추적, 성능 모니터링, 에러 로깅 시스템이 전혀 연동되어 있지 않습니다.

**문제점:**
*   사용자가 "앱 출시 알림 받기" 버튼을 얼마나 눌렀는지, 미션 참여 과정에서 어느 단계에 이탈했는지 데이터 기반의 의사결정을 내릴 수 없습니다.
*   프로덕션 환경에서 발생하는 클라이언트 사이드 에러를 인지할 방법이 없습니다.

**개선 제안:**
*   **에러 트래킹:** Sentry와 같은 프론트엔드 에러 모니터링 도구를 초기화하여 미처 잡지 못한 런타임 에러를 중앙 집중화하세요.
*   **이벤트 로깅:** 주요 CTA(Call To Action) 버튼 클릭, 모달 노출 등의 이벤트에 대해 Amplitude, Mixpanel 또는 Google Analytics를 연동하는 로깅 유틸리티를 구축하세요. 로깅 로직 역시 컴포넌트에 직접 하드코딩하기보다, 커스텀 훅이나 고차 컴포넌트(HOC)를 활용해 추상화하는 것이 좋습니다.

### 4. React Anti-patterns (리액트 안티패턴 및 DOM 제어)

**현상:**
`document.querySelectorAll`, `document.getElementById`를 사용해 DOM에 직접 접근하고 스타일(`e.target.style.opacity = '1'`)을 인라인으로 강제 변경하고 있습니다.

**문제점:**
*   React의 선언적(Declarative) 렌더링 패러다임과 가상 DOM(Virtual DOM) 생태계를 우회하는 심각한 안티패턴입니다. React가 상태를 업데이트할 때 DOM 상태와 충돌하여 렌더링 버그를 유발할 수 있습니다.

**개선 제안:**
*   **Refs 활용:** DOM 엘리먼트에 직접 접근해야 하는 경우 반드시 `useRef`를 사용하세요.
*   **선언적 스타일 제어:** DOM을 직접 조작하여 스타일을 바꾸지 말고, 상태(State)나 CSS 클래스(e.g., `isVisible` 상태에 따라 클래스네임 토글)를 통해 CSS/Tailwind 측에서 애니메이션을 처리하도록 위임해야 합니다. `scrollIntoView` 역시 대상 컴포넌트에 Ref를 달아 호출하는 방식으로 변경해야 합니다.

### 5. Scalability & State Management (확장성 및 상태 관리)

**현재 상태와 넥스트 스텝:**
현재는 정적 데이터와 `alert` 위주의 구성이지만, MVP가 발전함에 따라 실시간 미션 데이터(남은 시간, 참여자 수 동기화) 관리가 핵심이 될 것입니다. 상태 관리가 파편화되기 전에, 서버 상태(Server State)를 캐싱하고 동기화할 수 있는 **React Query (TanStack Query)** 같은 도구의 도입을 초기부터 고려하는 것을 권장합니다.
