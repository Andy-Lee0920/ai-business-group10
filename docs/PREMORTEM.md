# 미션팟 (NudgeMe) — Pre-mortem

> **작성일**: 2026-04-08
> **방법론**: Pre-mortem (Gary Klein) — "프로젝트가 이미 실패했다고 가정하고 원인을 거꾸로 추적"
> **시점 가정**: 베타 출시 후 3개월

---

## 0. 한 줄 요약 (Failure Headline)

> **"이 프로젝트는 3개월 내 실패했다. 단순한 성장 부진이 아니라, 금전 분배 구조의 규제 민감성, 악용 가능성에 비해 부족한 운영 역량, 그리고 플랫폼 수수료를 감안하지 않은 단위경제 부실이 동시에 발생했기 때문이다."**

세 가지 실패 축이 **동시에** 작동했다는 점이 핵심이다. 어느 하나만이었다면 회복 가능했으나, 세 축이 맞물려 자생력을 상실했다.

베타 출시 후 3개월 시점, 아래 중 **하나라도 충족 시 실패**로 간주한다.

| # | 실패 모드 | 측정 기준 | 영역 |
|---|---|---|---|
| **F1** | 사용자 미달 | MAU < 1,000 | 시장 / 획득 |

## 2. 세 가지 핵심 실패 축

### 🔴 축 A. 금전 분배 구조의 규제 민감성

미션팟은 **참가비를 모아 성공자에게 분배**하는 구조다. 이는 본질적으로 다음과 같은 법적 회색지대에 위치한다.

- **사행행위 등 규제 및 처벌 특례법** 저촉 가능성
  - "재물을 걸고 우연한 결과에 따라 재산상의 이익을 얻거나 잃는 행위"의 정의에 부분적으로 부합
  - 미션 성공이 "노력"인지 "우연"인지 모호한 사례 존재 (예: 날씨에 영향받는 산책 미션)
- **선불전자지급수단 발행 / 자금이체업** 인허가 이슈
  - 참가비 풀링 → 정산 구조가 전금법 규제 대상이 될 수 있음
- 베타 출시 시점에 **법무 검토 미완료** 상태로 강행할 경우, 출시 직후 민원/언론 한 건으로 서비스 중단 가능

**왜 실패로 이어지는가**: 한 번의 행정 지도 또는 언론 프레이밍("도박앱")으로 토스 앱인에서 강제 퇴출, 결제 채널 차단, 팀 신뢰도 붕괴까지 도미노로 진행된다.

---

### 🔴 축 B. 운영 역량 부족 (악용 대응 공백)

4인 팀 전원이 사이드 프로젝트로 참여, **풀타임 운영자 0명**.

악용 시나리오:
1. **인증 우회** — 타인 사진 도용, AI 생성 이미지, 과거 사진 재사용
2. **다계정 어뷰징** — 한 명이 여러 계정으로 환급 펌핑
3. **미션 매개 불법 행위** — "함께 걷기" 류 미션이 만남 빌미가 되어 사기 · 불법거래로 변질
4. **개인정보 노출** — 인증 사진에 타인 얼굴/번호판/주소 노출 → 프라이버시 분쟁
5. **CS 폭주** — 환급 지연 · 인증 거절 등 민원이 응답 SLA 없이 누적

**왜 실패로 이어지는가**: 위 사례 중 하나라도 SNS·커뮤니티에 노출되면, 사용자 1,000명 이하 단계에서는 회복이 불가능하다. 풀타임 인력이 없으므로 인시던트 발생 → 인지까지 평균 24시간+ 지연되고, 그 사이 피해가 확산된다.

---

### 🔴 축 C. 단위경제 부실 (수수료를 감안하지 않은 BM)

건당 1,000원 참가비 기준 추정:

| 항목 | 금액 (추정) |
|---|---|
| 참가비 (매출) | 1,000원 |
| 토스 앱인 / PG 수수료 (~3-4%) | -30~40원 |
| 환급 송금 수수료 | -수십~수백원 |
| 기부단체 송금 수수료 | -수십원 |
| 서버 · 인증 처리 원가 | -? |
| **건당 마진** | **사실상 0 또는 마이너스** |

본질적 수익원이 광고 / 스폰서 미션 / B2B 제휴여야 하는데, MAU 1,000 미달 상태에서는 이런 수익원을 유치할 수 없다. **닭과 달걀의 문제**.

**왜 실패로 이어지는가**: 사용자가 늘어날수록 손실이 늘어나는 구조 → 성장 자체가 자해(自害)가 된다.

## 3. 실패 모드별 원인 분해

### F1. MAU < 1,000
- 토스 앱인 노출 한정 (카테고리 경쟁 치열)
- "1,000원 참가비" 컨셉의 첫인상이 "이득"보다 "리스크"로 전달됨
- 친구 초대 보상 등 바이럴 장치 부재
- 페르소나 불명확 → 메시지가 누구에게도 꽂히지 않음

### F2. 수수료 ≥ 수익
- 사전 단위경제 시뮬레이션 누락
- 토스 앱인 수수료 정책을 출시 임박 시점에야 정확히 인지
- 광고 · 스폰서 미션 BM이 미구현 (MVP에서 빠짐)

### F3. 재접속 < 5회
- 미션 풀 빈약 → 같은 미션 반복 → 지루함
- 푸시 알림이 사용자 컨텍스트 무시
- 환급까지의 보상 지연 → 즉시성 부족
- 인증 절차 마찰 (GPS + 사진 + 타임스탬프)

### F4. 사행성 지적
- 출시 전 법무 검토 미완료
- "성공/실패에 따른 금전 분배" 구조 자체의 본질적 위험
- 약관 면책 조항 부실
- "도박앱" 프레임 확산 시 대응 매뉴얼 부재

### F5. 사기 · 불법 악용
- 인증 시스템의 어뷰징 내성 부족
- 다계정 탐지 로직 부재
- 미션 매개 오프라인 만남에 대한 가이드라인/차단 부재
- 신고 · 조치 워크플로우 부재

### F6. 운영 지속 불가
- 4인 모두 본업/학업 병행
- 한 명 이탈 시 전체 마비
- 인시던트 인지 → 대응까지 24h+ 지연
- 의사결정 속도 저하

## 5. 출시 전 필수 액션 (Pre-launch Checklist)

### A. 규제 (F4) — **출시 차단 항목**
- [ ] 법무법인 자문 1건 이상 (사행행위규제법 / 전금법 검토)
- [ ] 분배 구조를 "참가비 환급 + 잔여분 기부" 단일 트랙으로 단순화 (성공자 간 분배 금지)
- [ ] 약관 · 개인정보처리방침 검토 완료
- [ ] "도박앱" 프레임 대응 Q&A 문서 사전 작성

### B. 악용 방지 (F5) — **출시 차단 항목**
- [ ] 1인 1계정 정책 + 본인인증 (CI 기반)
- [ ] 인증 사진 메타데이터 검증 (촬영 시각, 위치, EXIF)
- [ ] 신고 · 차단 · 환불 워크플로우 문서화
- [ ] 미션 설계 가이드라인: 오프라인 만남 유발 미션 금지
- [ ] 인시던트 대응 SLA 정의 (최소 6시간 내 1차 대응)

### C. 단위경제 (F2) — **출시 차단 항목**
- [ ] 건당 손익 시뮬레이션 (수수료 모두 포함)
- [ ] 손익분기 MAU 산출
- [ ] 보조 수익원 1개 이상 확보 (스폰서 미션 LOI 등)
- [ ] 참가비 단가 재검토 (1,000원이 적정한가)

### D. 운영 (F6)
- [ ] 4인 역할 R&R 명문화 (특히 CS · 인시던트 담당)
- [ ] 온콜 로테이션 (야간 · 주말 포함)
- [ ] CS 채널 단일화 + 응답 SLA 공지
- [ ] 핵심 인력 1명 이탈 시나리오 문서화

### E. 리텐션 (F3)
- [ ] 미션 풀 최소 30종 확보 후 출시
- [ ] 환급 즉시 처리 (베타 기간 한정 자동 환급)
- [ ] 인증 절차 마찰 측정 (드롭률 모니터링)

### F. 획득 (F1)
- [ ] 페르소나 1개로 좁히고 메시지 통일
- [ ] 친구 초대 보상 메커니즘
- [ ] 베타 출시 전 웨이팅 리스트 500명 확보

## 6. Kill Criteria (사업 중단 기준)

다음 중 하나 발생 시, 추가 투입 없이 즉시 중단/피벗을 결정한다.

1. **법률**: 행정지도 · 시정명령 · 형사 입건 중 하나라도 발생
2. **악용**: 사용자 피해 사건 1건이라도 미디어/커뮤니티에 노출
3. **단위경제**: 출시 1개월 차에도 건당 마진 < 0
4. **운영**: 핵심 인력 2명 이상 이탈
5. **시장**: 출시 2개월 차 MAU < 300
